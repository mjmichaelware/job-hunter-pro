CONFIRMED IMPORTANT BUGS/FINDINGS: 1) Early logic falsely rejected valid food-service jobs as "not_food_service" because the negative keyword list used substring matching, e.g. "rn" could match inside ordinary words; v3 fixed this with exact token/phrase matching. 2) Google Places resolution initially accepted generic AI queries like “Salt Lake City hotel,” causing wrong place resolution, e.g. Grand America seasonal pool server resolved to Quality Inn. v3 improved place resolution by prioritizing explicit company/address queries. 3) Address extraction remained too aggressive and sometimes extracted sentence fragments like “13 has become one of Salt Lake City’s most recognized...” or “90 days of employment. Must...” as addresses. A v4 patch was proposed to make address extraction stricter and dedupe accepted jobs, but no successful confirmation is shown. 4) Duplicate jobs appeared, e.g. Cafe Staff duplicates from multiple sources. Dedupe should use canonical keys: title + normalized restaurant/place/address, not just job ID/source URL. 5) "lru_cache" was used on runtime functions ("geocode", "places_text_search", "place_details", "transit_to", "nearby_opportunities_cached", "serpapi_account_status"), but user correctly identified that Cloud Run instances scale to zero and in-process cache is not durable; it helps only within a warm container and should not be counted as persistent API savings. 6) Proposed scheduler with "INGEST_TOKEN" in URL is insecure because token appears in shell history, process list, Scheduler URI, and GCP console. Correct design is Cloud Scheduler HTTP target with OIDC service account allowed as Cloud Run invoker, and "/api/ingest" verifies OIDC, not URL token. 7) Manual metadata token access for GCS without caching/refresh logic is brittle; correct design uses "google-auth" "AuthorizedSession" so tokens refresh automatically. 8) "raw_job_queries()" calling nearby Places on every cold start causes latency and API usage; better: precomputed opportunity batches, Cloud Storage/DB cache, and scheduler ingestion. 9) SerpAPI budget is critical: user stated only about 200 SerpAPI searches left. Every 5 minutes is impossible if each batch uses SerpAPI. One week = 2016 five-minute intervals, so live SerpAPI every 5 minutes would burn quota instantly. Budget mode should use Google Places and stored history for broad browsing, SerpAPI only for controlled discovery batches.
