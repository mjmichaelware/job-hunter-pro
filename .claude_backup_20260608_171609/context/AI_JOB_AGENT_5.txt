Now imagine past the obvious. The system has a pipeline — discover, normalize, resolve_place, classify, score, dedupe, store, explain — and that pipeline can stream itself to the browser over Server-Sent Events. So build the pipeline as something the user can *watch breathe*: a chain of luminous nodes where each stage ignites the instant it begins real work, swells in proportion to the real count flowing through it, and visibly sheds the rejected results out the side with their actual rejection reasons labeled as they fall — not_food_service drifting one direction, outside_radius another, ambiguous_place_resolution oscillating before it drops. Make uncertainty visible and honest through type itself.
