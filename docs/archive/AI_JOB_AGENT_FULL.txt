Must...” as addresses. A v4 patch was proposed to make address extraction stricter and dedupe accepted jobs, but no successful confirmation is shown. 4) Duplicate jobs appeared, e.g. Cafe Staff duplicates from multiple sources. Dedupe should use canonical keys: title + normalized restaurant/place/address, not just job ID/source URL. 5) "lru_cache" was used on runtime functions ("geocode", "places_text_search", "place_details", "transit_to", "nearby_opportunities_cached", "serpapi_account_status"), but user correctly identified that Cloud Run instances scale to zero and in-process cache is not durable; it helps only within a warm container and should not be counted as persistent API savings. 6) Proposed scheduler with "INGEST_TOKEN" in URL is insecure because token appears in shell history, process list, Scheduler URI, and GCP console. Correct design is Cloud Scheduler HTTP target with OIDC service account allowed as Cloud Run invoker, and "/api/ingest" verifies OIDC, not URL token. 7) Manual metadata token access for GCS without caching/refresh logic is brittle; correct design uses "google-auth" "AuthorizedSession" so tokens refresh automatically. 8) "raw_job_queries()" calling nearby Places on every cold start causes latency and API usage; better: precomputed opportunity batches, Cloud Storage/DB cache, and scheduler ingestion. 9) SerpAPI budget is critical: user stated only about 200 SerpAPI searches left. Every 5 minutes is impossible if each batch uses SerpAPI. One week = 2016 five-minute intervals, so live SerpAPI every 5 minutes would burn quota instantly. Budget mode should use Google Places and stored history for broad browsing, SerpAPI only for controlled discovery batches.

## Section H — Markov Vacancy-Prediction (Beta Sub-Feature, Loosely Coupled)


Kept by request, but deliberately walled off from the core so it can never destabilize discovery. This is the one 3.0 idea with real upside for a job seeker: estimate which nearby restaurants are likely to have an opening SOON, before a listing is posted, and surface them as leads to apply to proactively.


### What it does
For each known place (from the Google Places opportunities you already collect), it watches signals over time and produces a single probability: likelihood of a near-term vacancy. High-probability places become proactive leads — distinct from real posted jobs, clearly labeled as predictions.


### Honest signals it can use (no fabrication)
- Review velocity changes — a sudden drop in review frequency or rating can correlate with staff churn.
- Hiring-history cadence — if a place posted openings repeatedly in past batches, it likely turns over on a rhythm.
- Recency since last seen hiring — time elapsed since a place last had a listing in your stored batches.
- Business status / new-location signals from Place Details.


### How it stays loosely coupled (the engineering rule)
- Lives in its own module: `predict/markov.py`, plus its own collection `predictions/{place_id}`. It imports FROM store/ and geo/ but nothing in the core imports it back.
- Its own endpoint: `/api/predictions`. The core dashboard works fully whether or not this endpoint or module exists.
- Its own UI tab labeled clearly as Beta / Predicted Leads — never mixed into Live Jobs or accepted results.
- Off by default behind a flag (`ENABLE_PREDICTIONS`). Flip it on only after the core is proven.
- Honest framing in the UI: predictions are heuristics from public signals, not facts. A predicted lead means "worth proactively contacting," not "a job exists."


### Where it slots in the timeline
Post-core, as P5 — after S12 and after the real free providers are added. It depends on having accumulated batch history (so the signals have data to work from), which only exists once the core has been running and storing batches for a while. Building it earlier would mean predicting from an empty dataset.


### What it must never do
- Never auto-apply to a predicted lead (same reason auto-apply is excluded everywhere).
- Never present a prediction as a real listing or inflate it into the accepted-jobs count.
- Never consume SerpAPI to generate predictions — it runs on already-stored data and free signals.
